Resource information

When you remove all resource information from an archive with ZipCleaner, this includes resource forks, as well as any .DS_Store files and custom icons for files and folders.

Traditionally, files on the Mac have had a different structure from those of other operating systems. Instead of treating a file simply as a named, ordered sequence of bytes, each Macintosh file has two separate sections or forks. One is known as the data fork and the other as the resource fork.
A file's resource fork contains the resources belonging to that file. If the file is an application, things like the menus, dialog boxes, icons, sounds, and information about the necessary memory for the program to run would be put there. Before the PowerPC architecture, the code to run the application also would always be stored in the resource fork. Document files, on the other hand, would seldom have information in their resource forks, but some programs would put things like the file's preference settings and window location in there. A file's data fork contains the data belonging to that file. It is the equivalent of what a file would be on another operating system.

As Mac OS was the only operating system that used resource forks, moving files to another OS would result in them losing their, sometimes essential, resource information. When OS X was introduced, Apple decided to move away from resource forks. Nowadays, only some files still use them. OS X still supports resource forks, though, e.g. saving them inside the zip archives the Finder creates. When you open such an archive on another OS, it will contain a folder named __MACOSX where the resource information of the uncompressed file(s) is stored.

By default, you can remove all resource information by simply dropping the archive on ZipCleaner. As removing resource forks can make files, (mostly older applications, but also some AppleScripts), unusable, ZipCleaner will warn you. You can tell it not do so, by changing the preferences.
